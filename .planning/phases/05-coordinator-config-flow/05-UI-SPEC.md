# Phase 5 — UI Design Contract

> Visual and interaction contract for the HA Config Flow UI.
> This phase has NO web frontend. All "UI" is the Home Assistant config flow,
> defined as Python schema + strings.json copy rendered by HA's built-in frontend.
> Generated by gsd-ui-researcher.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — HA config flow (Python schema + strings.json) |
| Preset | not applicable |

| Icon library | not applicable — HA renders field icons automatically |

## Config Flow Schema

This is the primary "visual contract" for Phase 5: what fields exist, their types, defaults,
and validation constraints.

### Step: user (only step)

| Field | Type | Default | Placeholder | Validation |
|-------|------|---------|-------------|------------|
| `host` | `str` | none | `192.168.x.x` | Non-empty, stripped of whitespace |

**Hidden fields (not shown to user):**

| Field | Source | Value |
|-------|--------|-------|
| `port` | `const.py` `DEFAULT_PORT` | `8889` |
| `poll_interval` | `const.py` `DEFAULT_POLL_INTERVAL` | `30` |

**Schema declaration (Python):**

```python
vol.Schema({
    vol.Required(CONF_HOST): str,
})
```

**Note:** Port and poll interval are not exposed in the config flow UI.
Options Flow (to reconfigure these) is explicitly deferred to v2 per REQUIREMENTS.md CONF-10.

## Interaction States

| State | Trigger | HA Mechanism | User Experience |
|-------|---------|--------------|-----------------|
| Initial | User opens config flow | `async_step_user` called with no input | Form shown with `host` field empty |
| Submitting | User enters IP and clicks Submit | `async_step_user` called with `user_input` | HA shows spinner; flow runs connection test |
| Connection error | `luxtronik` read raises any exception | Return `errors={"base": "cannot_connect"}` | Form re-shown with error message below the host field |
| BenPru conflict | `hass.config_entries.async_entries("luxtronik")` finds matching host | Return `self.async_abort(reason="already_configured_benPru")` | Abort screen shown (no form) |
| Duplicate own entry | `await self.async_set_unique_id(host)` finds match | `self._abort_if_unique_id_configured()` | Abort screen shown (no form) |
| Success | Connection test passes, no conflicts | `self.async_create_entry(title=host, data={...})` | Integration created; HA navigates to integration page |

## Copywriting Contract

All copy lives in `strings.json` (source of truth) and is mirrored in `translations/en.json`.
German translation (`translations/de.json`) is deferred to Phase 7 (HACS-02).
A `strings.json` stub for config flow labels is included in this phase per CONTEXT.md D-11 discretion.

### strings.json structure

```json
{
  "config": {
    "step": {
      "user": {
        "title": "Connect to Luxtronik 2.0",
        "description": "Enter the IP address of your heat pump controller.",
        "data": {
          "host": "IP Address"
        },
        "data_description": {
          "host": "Local IP address of the Luxtronik 2.0 controller (e.g. 192.168.x.x)"
        }
      }
    },
    "error": {
      "cannot_connect": "Cannot connect to the heat pump. Check the IP address and make sure the controller is reachable on port 8889."
    },
    "abort": {
      "already_configured": "This heat pump is already configured.",
      "already_configured_benPru": "A BenPru/luxtronik integration is already connected to this IP address. Running both integrations simultaneously will cause connection conflicts. Remove the luxtronik integration first."
    }
  }
}
```

### Copywriting contract table

| Element | Copy |
|---------|------|
| Config flow title | Connect to Luxtronik 2.0 |
| Step description | Enter the IP address of your heat pump controller. |
| `host` field label | IP Address |
| `host` field description | Local IP address of the Luxtronik 2.0 controller (e.g. 192.168.x.x) |
| Primary CTA label | Submit (HA default — not overridden) |
| `cannot_connect` error | Cannot connect to the heat pump. Check the IP address and make sure the controller is reachable on port 8889. |
| `already_configured` abort | This heat pump is already configured. |
| `already_configured_benPru` abort | A BenPru/luxtronik integration is already connected to this IP address. Running both integrations simultaneously will cause connection conflicts. Remove the luxtronik integration first. |

**Copy rules:**
- All copy in US English (source strings.json)
- No technical jargon in user-facing messages: write "heat pump controller" not "Luxtronik binary TCP endpoint"
- Error messages follow the pattern: [what went wrong] + [what to do next]
- Abort messages explain the conflict and the resolution path
- Use sentence case for all labels and descriptions (not Title Case)

## Implementation Notes for Executor

1. `config_flow.py` must subclass `config_entries.ConfigFlow` with `DOMAIN = DOMAIN` and `VERSION = 1`
2. `async_step_user` performs the connection test inline using `hass.async_add_executor_job`
3. Connection test: instantiate `luxtronik.Luxtronik(host, DEFAULT_PORT)`, call `read()` in executor, then discard — same connect-per-call pattern as `LuxtronikClient.async_read()`
4. Wrap the executor call in `try/except Exception` — catch all errors and map to `cannot_connect`
5. Use `await self.async_set_unique_id(host)` then `self._abort_if_unique_id_configured()` for duplicate detection before the BenPru check
6. BenPru check: `hass.config_entries.async_entries("luxtronik")` — check each entry's `data.get("host")` against the submitted host
7. `strings.json` stub must be committed in this phase even if translations are deferred — HA validates `strings.json` at HACS check time
