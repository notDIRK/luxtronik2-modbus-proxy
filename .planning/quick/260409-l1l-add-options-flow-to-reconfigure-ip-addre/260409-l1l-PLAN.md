  <name>Task 2: Wire update listener in __init__.py for entry reload on options change</name>
  <files>custom_components/luxtronik2_modbus_proxy/__init__.py</files>
  <action>
Modify `async_setup_entry` to register an update listener that reloads the config entry
when the user saves new options:

1. After `await coordinator.async_config_entry_first_refresh()` and before storing the
   coordinator in `hass.data`, add:
   ```python
   entry.async_on_unload(entry.add_update_listener(_async_update_listener))
   ```

2. Add a module-level async function:
   ```python
   async def _async_update_listener(hass: HomeAssistant, entry: ConfigEntry) -> None:
       """Reload the config entry when options change.

       Triggered by the Options Flow saving new values (host or poll_interval).
       A full reload recreates the coordinator with the updated host/port and
       poll interval, ensuring the new settings take effect immediately.
       """
       await hass.config_entries.async_reload(entry.entry_id)
   ```

This is the standard HA pattern: when the options flow calls async_update_entry,
HA fires the update listener, which triggers a full unload+setup cycle. The new
setup reads the updated data dict (with new host/poll_interval) and creates a fresh
coordinator pointing at the correct IP with the correct interval.

Note: Since we store poll_interval in entry.data (not entry.options), and the options
flow uses async_update_entry to modify entry.data, the update listener fires on any
data change. The reload approach is correct here because a host change requires a new
coordinator anyway, and a poll_interval change is rare enough that a brief reload is
acceptable.
